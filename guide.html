ภาพรวมระบบ

ระบบนี้เป็น Industrial IoT Data Platform แบบ Airgap (ไม่ต้องใช้อินเทอร์เน็ต) ทำงานบน Kubernetes (k3s) single-node เพื่อเก็บ วิเคราะห์ และแสดงผลข้อมูลจาก Sensor อุตสาหกรรม 12 ตัว แบบ Real-time และเก็บใน Data Lake สำหรับการวิเคราะห์ระยะยาว

Real-time Monitoring

OPC Simulator → Kafka → Telegraf → InfluxDB → Grafana
Refresh ทุก 5 วินาที

Data Lake & Analytics

Kafka → NiFi Core → MinIO → Trino
Partition by year/month/day, NDJSON format

Asset Management

OpenMAINT CMMS บน PostgreSQL 15
ติดตาม อุปกรณ์ + Maintenance

📨 Kafka (KRaft) DMZ Zone — Port :32092

Message Broker หลักที่คั่นระหว่าง OT Zone กับ IT Zone เพื่อความปลอดภัยและ buffer ข้อมูล ทำงานในโหมด KRaft (ไม่ต้องใช้ ZooKeeper แยก)

TopicMessagesConsumerLAG

opc-metrics~42,368telegraf-opc-consumer≈ 5

ℹ️
LAG ≈ 5 หมายถึง Telegraf consume ช้ากว่า producer 5 messages — ถือว่าปกติมาก หาก LAG เพิ่มต่อเนื่องให้ตรวจสอบ Telegraf pod

📊 Telegraf IT Zone

Agent ที่ consume topic opc-metrics แล้วแปลงข้อมูล JSON → InfluxDB Line Protocol และเขียนลง InfluxDB อัตโนมัติ ไม่มี UI ใช้การ monitor ผ่าน logs

# Telegraf config สำคัญ
[[inputs.kafka_consumer]]
  brokers = ["kafka.dmz.svc:9092"]
  topics = ["opc-metrics"]
  consumer_group = "telegraf-opc-consumer"
  data_format = "json"

[[outputs.influxdb_v2]]
  urls = ["http://influxdb.it.svc:8086"]
  token = "$INFLUX_TOKEN"
  bucket = "opc-data"

🔧 NiFi Core IT Zone — Port :31443

Consumer ของ opc-datalake ทำหน้าที่ batch + save ข้อมูลลง MinIO แบบ partitioned

Step 1
ConsumeKafkaRecord
อ่านจาก topic opc-datalake
Step 2
MergeRecord
รวม 100 records เป็นไฟล์เดียว
Step 3
UpdateAttribute
ตั้ง path: data/year=YYYY/month=MM/day=DD/
Step 4
PutS3Object
เขียนไฟล์ NDJSON ลง MinIO (S3 API)
⚠️
NiFi Core ใช้ RAM สูงสุดในระบบ ≈ 2,001Mi หากพบ OOMKilled ให้เพิ่ม memory limit หรือลด MergeRecord batch size

🪣 MinIO IT Zone — Port :30901

Object Storage เข้ากันได้กับ AWS S3 API เก็บไฟล์ NDJSON แบบ Hive-style partitioning

opc-raw/
└── data/
    └── year=2026/
        └── month=05/
            └── day=04/
                ├── part-0001.ndjson  (100 records)
                ├── part-0002.ndjson
                └── ...  ← 13 files วันนี้

⚡ Trino IT Zone — Port :30800

Distributed SQL Query Engine อ่านข้อมูลจาก MinIO โดยตรง ไม่ต้อง ingest ข้อมูลใหม่ ใช้ SQL มาตรฐาน

-- ตัวอย่าง query วิเคราะห์ข้อมูล
SELECT year, month, day, COUNT(*) records
FROM minio.opc.sensor_data
GROUP BY ALL ORDER BY year, month, day;

Phase 1 — Infrastructure (Namespace, StorageClass, PV)

# สร้าง Namespaces
kubectl create namespace dmz
kubectl create namespace it

# StorageClass (local-path หรือ hostPath)
cat <<EOF | kubectl apply -f -
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: local-storage
provisioner: kubernetes.io/no-provisioner
volumeBindingMode: WaitForFirstConsumer
EOF

# PersistentVolumes (ตัวอย่าง Kafka)
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-kafka
spec:
  capacity:
    storage: 50Gi
  accessModes: [ReadWriteOnce]
  storageClassName: local-storage
  hostPath:
    path: /data/kafka
EOF

# สร้าง directories บน host
sudo mkdir -p /data/{kafka,minio,influxdb,nifi-core,openmaint}

4. เมื่อเกิดปัญหา — วิธีตรวจสอบ

✅
เริ่มต้นตรวจสอบ: รัน kubectl get pods -A เพื่อดูสถานะ pods ทั้งหมดก่อนเสมอ
🔴 Pod ไม่ขึ้น / CrashLoopBackOff ▼ 
# ดูสถานะ pods ทั้งหมด
kubectl get pods -A -o wide

# ดู log ของ pod ที่มีปัญหา
kubectl logs -n it <pod-name> --previous
kubectl logs -n it <pod-name> --tail=50

# ดู events (มักบอกสาเหตุ)
kubectl describe pod -n it <pod-name> | tail -30

# กรณี OOMKilled — เพิ่ม memory limit

  --limits=memory=4Gi --requests=memory=2Gi
📭 Kafka Consumer LAG สูงผิดปกติ ▼ 
# ตรวจสอบ consumer group lag
kubectl exec -n dmz kafka-0 -- kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe --all-groups

# ตรวจสอบ Telegraf ยังทำงานอยู่ไหม
kubectl logs -n it deploy/telegraf --tail=20
kubectl top pod -n it | grep telegraf

# Restart Telegraf หาก stuck
kubectl rollout restart deployment/telegraf -n it

# ตรวจสอบ NiFi Core lag
kubectl logs -n it <nifi-core-pod> | grep -i "kafka\|lag\|error"
⚠️
LAG ปกติ ≈ 2–10 | LAG > 1000 = มีปัญหา consumer หยุดทำงาน

5. Tips · ข้อควรระวัง · อะไรที่ดีกว่า

✅ Best Practices

  • → ใช้ KRaft mode (ไม่ต้องใช้ ZooKeeper แยก)
  • → ตั้ง retention policy ให้ Kafka (7d) เพื่อไม่ให้ disk เต็ม
  • → batch MinIO ทุก 100 records ลด file overhead
  • → ใช้ NDJSON (line-delimited JSON) อ่านง่ายกว่า nested JSON
  • → Hive partitioning year/month/day ช่วย Trino query เร็วขึ้นมาก

⚠️ ข้อควรระวัง

  • → NiFi Edge/Core ใช้ HTTPS self-signed cert → browser จะ warn
  • → AKHQ ไม่มี auth → อย่า expose ออก internet
  • → OpenMAINT init DB นาน 3-5 นาที อย่า restart ระหว่างนั้น
  • → Trino partition ไม่ auto-sync ต้อง CALL sync_partition_metadata
  • → NiFi Core RAM สูง (2GB) → monitor OOM เสมอ

🚀 อะไรที่ดีกว่า (Future Improvements)

  • → แทน NiFi Edge ด้วย Telegraf OPC-UA plugin (เบากว่ามาก)
  • → แทน NDJSON ด้วย Parquet format → Trino query เร็ว 5-10x
  • → เพิ่ม Apache Iceberg แทน Hive → ACID transactions
  • → ใช้ Kafka Schema Registry → ป้องกัน schema drift
  • → เพิ่ม AlertManager + PagerDuty สำหรับ Grafana alerts

🔒 Security Warnings

  • → เปลี่ยน default password admin/admin ของ OpenMAINT ทันที
  • → เพิ่ม Network Policy แยก namespace ot/dmz/it
  • → ใช้ Kubernetes Secrets แทน hardcode credentials ใน env
  • → เปิด TLS สำหรับ Kafka ก่อน expose :32092 ออกภายนอก

CronJob — Auto Partition Sync (แนะนำ)

cat <<EOF | kubectl apply -f -
apiVersion: batch/v1
kind: CronJob
metadata:
  name: trino-partition-sync
  namespace: it
spec:
  schedule: "5 0 * * *"  # ทุกวัน 00:05
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: trino-sync
            image: trinodb/trino:latest
            command:
            - trino
            - --server=http://trino.it.svc:8080
            - --execute
            - |
              CALL minio.system.sync_partition_metadata(
                schema_name=>'opc',
                table_name=>'sensor_data',
                mode=>'ADD'
              );
EOF
