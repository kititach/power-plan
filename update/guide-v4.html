1. ภาพรวม + สถานะปัจจุบัน

Project นี้คืออะไร

ระบบ Industrial Data Platform สำหรับโรงงาน — รับข้อมูลจาก OPC UA Server (300+ sensor) ส่งผ่าน Kafka แล้วกระจายไปยัง 3 ปลายทาง:

  • InfluxDB + Grafana — real-time monitoring dashboard

คำอธิบายแต่ละ Block

① Prosys OPC UA Server

OPC UA simulation server รันบนเครื่อง mintserver (10.85.3.100) ผ่าน systemd prosys-opc — exposes 307 simulated industrial tags (Temp, Pressure, Flow, Vibration, Current, ฯลฯ) บน port 53530 ผ่าน opc.tcp protocol

หน้าที่: จำลอง PLC/SCADA จริง เพื่อทดสอบ pipeline ก่อนเชื่อมต่อ OPC UA จริงในโรงงาน

② NiFi Edge

Apache NiFi รันใน k3s ns it — ใช้ Eclipse Milo OPC UA client (Java library) + custom Groovy script (opc_reader_final.groovy) อ่านค่า 307 tags ทุก 2 วินาที

Output format: JSON แบบ flat — 307 field ระดับ top-level + metadata (timestamp, source_id, tag_count, bad_count) → ส่งเข้า Kafka topic opc-raw-data

Key config: PublishKafka.Transactions Enabled = false

③ Apache Kafka

ใช้ Strimzi operator deploy Kafka แบบ KRaft mode (ไม่ต้องใช้ Zookeeper) — single broker, 3 partitions — ทำหน้าที่เป็น buffer + fan-out layer ระหว่าง producer (NiFi Edge) กับ consumer 3 ตัวที่ทำงานขนานกัน

ทำไมใช้ Kafka: ถ้า consumer ตัวใดตัวหนึ่งล้ม producer ไม่เดือดร้อน, ผูก/แตก consumer ใหม่ได้ทันที, ข้อมูลย้อนหลังอ่านได้ภายใน retention window (7 วัน)

④A Path A — Real-time (Telegraf → InfluxDB → Grafana)

Telegraf consumer group telegraf-opc-consumer อ่าน Kafka → parse JSON ผ่าน tag_keys → เขียนเข้า InfluxDB v2 measurement opc_data (bucket opc-data)

Grafana query Flux ที่ datasource InfluxDB-OPC → แสดง real-time dashboard 12 panels (lag < 5 msg ปกติ)

④B Path B — Data Lake (NiFi Core → MinIO → Trino)

NiFi Core consumer group nifi-core-consumer — ConsumeKafka → MergeRecord รวม 100 messages/file → PutS3Object ไป MinIO bucket opc-raw (Parquet-ready folder structure year=YYYY/month=MM/day=DD/)

Trino query เป็น SQL ผ่าน partition ที่ CronJob trino-partition-sync อัปเดต 2x/วัน (00:05, 12:05)

④C Path C — Asset Management (OpenMAINT Bridge → OpenMAINT)

Python script tools/openmaint_bridge.py รันเป็น systemd service บน host (ไม่ใช่ใน k8s) — consumer group openmaint-consumer อ่าน Kafka → ตรวจค่าทุก tag กับ 19 threshold rules (cooldown 5 นาที/tag กัน spam)

เมื่อ violation: สร้าง Alarm ใน OpenMAINT + CorrectiveMaint work order อัตโนมัติ (มอบหมายช่างได้ทันที)

ทำไม systemd ไม่ใช่ pod: ใช้ Python venv local + Kafka access ตรง — ไม่ต้องสร้าง container image เพิ่ม, debug ง่าย, restart เร็ว

7. OPC UA + Prosys Server

OPC UA คืออะไร

OPC UA (Unified Architecture) คือ standard สื่อสารระหว่าง OT equipment (PLC, SCADA, sensor) กับ IT systems — เป็น protocol ที่ใช้กันทั่วโลกในโรงงานอุตสาหกรรม

Polling Model ปัจจุบัน

NiFi Edge ใช้ Eclipse Milo (Java OPC UA library) เรียก readValues() ทุก 2 วินาที อ่านค่า 307 tags ในรอบเดียว (~0.3-0.5 วินาที)

NiFi Edge (Groovy)
   │ readValues([nodeId1, nodeId2, ..., nodeId307])
   ▼
Prosys OPC UA Server (mintserver:53530)
   │ ส่งค่ากลับใน 1 response
   ▼
NiFi Edge รับ → flatten → publish Kafka

Prosys Server บน mintserver

รันเป็น systemd service ชื่อ prosys-opc

# SSH เข้า mintserver
ssh demo@10.85.3.100   # password: @123456789

# คำสั่งจัดการ
sudo systemctl status prosys-opc       # ดูสถานะ
sudo systemctl start prosys-opc        # เริ่ม
sudo systemctl stop prosys-opc         # หยุด
sudo systemctl restart prosys-opc      # รีสตาร์ท
sudo journalctl -u prosys-opc -f       # ดู log realtime
ทำไมต้องใช้ wrapper script: Prosys ใช้ install4j launcher ที่ fork Java แล้ว exit ทันที ทำให้ Type=simple ของ systemd คิดว่า service จบแล้ว — wrapper script /usr/local/bin/prosys-start.sh ใช้ kill -0 loop รอ Java process แทน wait

Kafka Fan-out: Consumer แต่ละ group มี offset แยกกัน — เพิ่ม consumer ใหม่ (เช่น openmaint-consumer) ไม่กระทบ consumer เดิม

คำสั่ง Kafka ที่ใช้บ่อย

# List topics

  bin/kafka-topics.sh --bootstrap-server localhost:9092 --describe --topic opc-raw-data

# ดู offset (rate การไหลของข้อมูล)
kubectl exec -n dmz kafka-cluster-broker-0 -c kafka -- \
  bin/kafka-get-offsets.sh --bootstrap-server localhost:9092 --topic opc-raw-data

# ดู consumer lag
kubectl exec -n dmz kafka-cluster-broker-0 -c kafka -- \
  bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --describe --all-groups

# Consume 1 message ดู
kubectl exec -n dmz kafka-cluster-broker-0 -c kafka -- \
  bin/kafka-console-consumer.sh --bootstrap-server localhost:9092 \
  --topic opc-raw-data --max-messages 1 --timeout-ms 5000

13. OpenMAINT (CMMS)

OpenMAINT คืออะไร

OpenMAINT = Computerized Maintenance Management System (CMMS) สำหรับจัดการ asset, work order, preventive maintenance, spare parts สร้างจาก CMDBuild 3.4.1 framework

เข้าใช้งาน

URL: http://10.85.3.104:30885/cmdbuild (admin / admin)

โครงสร้างเมนูหลัก

เมนูใช้ทำอะไร
Facilities and assetsจัดการ asset (Boiler, Pump, Motor, Building)
Maintenance managementงานบำรุงรักษา
↳ Corrective maintenanceWork order แก้ไขฉุกเฉิน ⭐ (bridge สร้างให้อัตโนมัติ)
↳ Preventive maintenanceแผนบำรุงรักษาตามกำหนด
↳ AlarmsAlarm log ⭐ (bridge สร้างให้อัตโนมัติ)
InventorySpare parts, stock
ConfigurationsSetup users, lookup, workflow

CorrectiveMaint Workflow

Opening → Assignment → Management → Estimate → Control → Execution → Accounting → Closure
   ↑          ↑                                              ↑              ↑
bridge    advance อัตโนมัติ                          ช่างซ่อม         ปิดงาน
สร้าง    (ผ่าน _advance: True)                    update status   ใส่ค่าใช้จ่าย

การ Advance Activity

  1. คลิก work order → กด ✏️ Edit
  2. กรอกข้อมูล required (Site, Category, Team, Priority)
  3. กด Execute (มุมขวาล่าง) → workflow เลื่อนไป step ถัดไป
DMS Disabled: เราปิด CMIS/Alfresco DMS ไปแล้ว (ไม่มีในระบบ airgap) ใน _SystemConfig table set org.cmdbuild.dms.enabled = false ก่อนหน้านี้ทำให้ Execute step ขึ้น "Generic error"

14. OpenMAINT Bridge ⭐

ใหม่ 2026-05-08: Python service เชื่อม Kafka → OpenMAINT — เมื่อ sensor ผิดปกติ สร้าง Alarm + Work Order อัตโนมัติ พร้อม cooldown 5 นาที/tag กัน spam

Architecture

Kafka: opc-raw-data
  └─► openmaint-bridge (systemd, on host)
        ├─ kafka-python consumer group: openmaint-consumer
        ├─ ตรวจ 19 threshold rules
        ├─ Cooldown 300s/tag
        └─► REST API → OpenMAINT
              ├─ POST /classes/Alarm/cards
              └─ POST /processes/CorrectiveMaint/instances
                    (with _advance: True → ผ่าน Opening ไป Assignment)

ทำไมรันบน host ไม่ใช่ k3s pod

ระบบเป็น airgap — ไม่มี Python image ใน k3s containerd แต่ host มี Python 3.12 + venv ที่มี kafka-python อยู่แล้ว จึง deploy เป็น systemd service บนเครื่อง mintpower ตรงๆ

17. Troubleshooting (ปัญหาที่เคยเจอ)

🔴 OPC data หยุดไหลเข้า Kafka

  1. เช็ค Prosys: ssh demo@10.85.3.100 "sudo systemctl status prosys-opc"
  2. เช็ค NiFi Edge log: kubectl logs -n it -l app=nifi-edge --since=5m | grep -i "opc\|error"
  3. เช็ค disk NiFi: kubectl exec -n it deployment/nifi-edge -- df -h /opt/nifi
  4. Restart NiFi Edge: kubectl rollout restart deployment/nifi-edge -n it

🔴 OpenMAINT ล่ม / ตอบ Connection refused

อาการ: pod READY 1/1 แต่ curl ไม่ตอบ, RAM ใช้แค่ 2Mi

สาเหตุ: Tomcat process ตายใน container แต่ entrypoint script ยังรันอยู่ (sleep loop)

# เช็ค process ใน pod
kubectl exec -n it deployment/openmaint -- ps aux

# ถ้าเห็นแค่ sleep 15 — Tomcat ตาย
kubectl rollout restart deployment/openmaint -n it

# รอ ~5 นาที
until curl -s "http://10.85.3.104:30885/cmdbuild/services/rest/v3/boot/status" | grep -q READY; do
  sleep 10
done

🟡 Grafana account locked

อาการ: "too many consecutive incorrect login attempts"

อย่า restart pod! lockout เก็บใน SQLite PVC — restart ไม่ช่วย แต่ทำให้ datasource token หายแทน

วิธีแก้:

# exec เข้า pod
GRAFANA_POD=$(kubectl get pods -n it -l app=grafana -o jsonpath='{.items[0].metadata.name}')
kubectl exec -n it $GRAFANA_POD -- \
  sqlite3 /var/lib/grafana/grafana.db "DELETE FROM login_attempt;"

🟡 Grafana datasource token หาย

ปกติ postStart hook ตั้ง token ให้แล้ว ถ้าไม่ทำงาน:

INFLUX_TOKEN=$(kubectl get secret -n it influxdb-secret -o jsonpath='{.data.admin-token}' | base64 -d)
curl -s -u "admin:admin2026" \
  -X PUT -H "Content-Type: application/json" \
  -d "{\"id\":1,\"uid\":\"influxdb-opc\",...,\"secureJsonData\":{\"token\":\"$INFLUX_TOKEN\"}}" \
  "http://10.85.3.104:30300/api/datasources/1"

🔴 Trino ไม่เห็น partition ใหม่

อาการ: วันนี้มีไฟล์ใน MinIO แต่ Trino query ไม่เห็น

แก้: trigger sync manual

kubectl create job trino-sync-manual-$(date +%Y%m%d%H%M) -n it \
  --from=cronjob/trino-partition-sync

🟡 Bridge ขาดการเชื่อมต่อ Kafka

# เช็ค consumer group active
kubectl exec -n dmz kafka-cluster-broker-0 -c kafka -- bash -c \
  "bin/kafka-consumer-groups.sh --bootstrap-server localhost:9092 --describe --group openmaint-consumer"

# ถ้า "Inactive" → restart bridge

18. Backup & Recovery

⚠️ ปัจจุบันไม่มี backup อัตโนมัติ — ทุกอย่างเก็บใน NVMe local ถ้า disk เสีย = ข้อมูลหาย ควรเพิ่ม backup ในแผน (ดู Section 19)

Backup Manual

1. OpenMAINT PostgreSQL

kubectl exec -n it deployment/postgres-openmaint -- \
  pg_dump -U postgres openmaint | gzip > openmaint-$(date +%Y%m%d).sql.gz

2. InfluxDB

INFLUX_TOKEN=$(kubectl get secret -n it influxdb-secret -o jsonpath='{.data.admin-token}' | base64 -d)
kubectl exec -n it deployment/influxdb -- \
  influx backup --token "$INFLUX_TOKEN" --org mintpower-org /tmp/backup
kubectl cp it/$(kubectl get pod -n it -l app=influxdb -o jsonpath='{.items[0].metadata.name}'):/tmp/backup ./influx-backup-$(date +%Y%m%d)

3. NiFi Flow

# flow.json.gz อยู่ใน PVC
kubectl cp it/$(kubectl get pod -n it -l app=nifi-edge -o jsonpath='{.items[0].metadata.name}'):/opt/nifi/nifi-current/conf/flow.json.gz ./nifi-edge-flow-$(date +%Y%m%d).json.gz

4. Grafana SQLite

kubectl exec -n it deployment/grafana -- \
  sqlite3 /var/lib/grafana/grafana.db ".backup /tmp/grafana.db.bak"
kubectl cp it/$(kubectl get pod -n it -l app=grafana -o jsonpath='{.items[0].metadata.name}'):/tmp/grafana.db.bak ./grafana-$(date +%Y%m%d).db

Glossary ศัพท์เทคนิค

คำคำอธิบาย
airgapสภาพแวดล้อมที่ไม่มีอินเทอร์เน็ต — image และ package ต้อง import ล่วงหน้า
BPMN / Bonitaworkflow engine ที่ CMDBuild ใช้จัดการ process (Opening → Assignment → ...)
CMDBuildOpen-source CMDB framework ที่ OpenMAINT สร้างต่อจากมัน
CMISContent Management Interoperability Services — ใช้คุยกับ Alfresco DMS (เราปิดไป)
CMMSComputerized Maintenance Management System — ระบบจัดการงานซ่อมบำรุง
CRDCustom Resource Definition — k8s API ที่ third-party operator เพิ่มเข้ามา (เช่น KafkaTopic)
DMSDocument Management System — ในระบบนี้ไม่มี Alfresco/CMIS
DMZDemilitarized Zone — namespace กลางระหว่าง OT/IT (Kafka อยู่นี่)
Eclipse MiloJava OPC UA client/server library ที่ NiFi Edge ใช้
Fan-out1 producer กระจายไปหลาย consumer — Kafka ทำได้โดย consumer group แยกกัน
Flat JSONJSON ที่ไม่มี nested object — ทุก field อยู่ top-level (Telegraf ชอบ)
FluxInfluxDB v2 query language
FlowFileหน่วยข้อมูลใน NiFi (= 1 record + metadata)
GroovyJVM scripting language ที่ NiFi ExecuteGroovyScript รัน
install4jJava launcher ที่ Prosys ใช้ — fork process แล้ว exit (ทำให้ systemd งง)
KRaftKafka mode ใหม่ — ไม่ต้องใช้ Zookeeper
OPC UAOpen Platform Communications Unified Architecture — standard อุตสาหกรรม
ParquetColumnar storage format — เล็กกว่า JSON ~8 เท่า, query เร็วกว่า
postStart hookk8s lifecycle event — รันคำสั่งทันทีหลัง container start
ProsysOPC UA simulation server (Java) ที่เราใช้สร้าง 307 tags
PVC / PVPersistentVolumeClaim / PersistentVolume — storage k8s
StrimziKafka operator ที่ deploy Kafka บน k8s
systemdservice manager ของ Linux — เราใช้กับ prosys-opc + openmaint-bridge
TelegrafMetrics collector — input Kafka, output InfluxDB
TSMTime-Structured Merge tree — InfluxDB storage engine
